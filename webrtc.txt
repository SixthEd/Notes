NAT: Network Address Translation
If we have public ip address then we can connect to that device using like aws ec2 
We are connected to a router or wifi so we don't have any public ip address so router has both public ip address and private ip address (gateway)

we send the request using router ip to other devices

NAT Method: 
1) One to One NAT (Full-cone NAT)
2) Address restricted NAT
3) Port Restricted NAT
4) Symmetric NAT // Not used in Webrtc

Stun server:
Session Traversal Utilities for NAT
Tell myip address/port through NAT
Works for Full-cone, Port/Address restricted NAT
Doesn't works for symmetric NAT
STUN servers port 3478, 5349 for TLS
Cheap to maintain because it works on docker container that has stun server 

TURN:
Traversal using relays around NAT
In case symmetric NAT we use TURN
It's just a server that relays packets
TURN default server port 3478, 5349 for TLS
Expensive for maintain and run

ICE:
It is procotocol
Interactive Connectivity Enstablishment
Collect all ip address, private ip address, reflexive ip address and these are called ICE Candidates (possibilities of things that can you communicate through)and sent to the remote peer via SDP (Session Description protocols)

SDP: (offer)
Session Description Protocols 
A format that Describes ice candidates, networking, options, media options, security options and other stuff
Not really a protocol its a format
The goal is to take the SDP generated by user and send to it "somehow" to the other party


Webrtc Demystified :

1. A wants to connect to B
2. A creates an "offer" , if finds all the ice candidates , security options, audio/video options and generates SDP, the offer is basically the SDP
3. A signals the offer somehow to B  (whatsapp)
4. B creates the "answer" after setting A's offer
5. B signals the "answer" to A
6. Connection is created.
